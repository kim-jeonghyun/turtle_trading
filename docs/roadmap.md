## 8. 비판적 반론과 합의

### Q1: Paper Trading이 정말 필요한가?
- **반론**: 소액(100만원)으로 바로 실거래하면 실제 피드백이 빠름
- **합의**: 별도 인프라 대신 기존 `dry_run` 확장으로 축소. v3.8.0의 서브태스크로 포함

### Q2: 일별 OHLCV 축적이 왜 필요한가?
- **반론**: yfinance 온디맨드로 충분하지 않은가?
- **합의**: yfinance는 장애/제한 가능. KOSPI 200 + KOSDAQ 150 (~350종목) 축적. API 18초/일, 10MB/년으로 오버헤드 최소이면서 유니버스 확장/스크리닝 기반 확보

### Q3: 킬 스위치가 정말 중요한가?
- **반론**: 이미 2N 스톱로스 + 포지션 한도가 있음
- **합의**: **절대 필수**. 2N은 개별 포지션 보호. 시스템 전체 오작동(API 버그, 코드 에러)에는 킬 스위치만 대응 가능

### Q4: 유니버스 자동 갱신이 Phase 2인가?
- **반론**: 수동 관리가 충분하지 않은가?
- **합의**: 1인 개발/~30종목에선 수동 관리 충분. Phase 2 후반으로 연기

### Q5: 1인 개발자에게 현실적인가?
- **반론**: 범위를 줄여야 하는 것 아닌가?
- **합의**: 현실적. v3.2→v3.6 (5개 마일스톤)을 ~2주에 완료한 실적. 각 마일스톤 5개 이내 이슈로 제한하면 1-2주/마일스톤 달성 가능
